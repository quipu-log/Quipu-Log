# quipu-core

[English](README.md) | 한국어

[Quipu-Log](../../README.ko.md)의 임베디드 저장·질의 코어입니다. append-only 해시 체인 세그먼트, 버전 관리되는 엔티티 레지스트리, 필드 단위 보호, 보존 정책, 시점 질의를 담당합니다. 이 크레이트는 동기 엔진입니다. 비동기 파이프라인과 HTTP 프록시는 `quipu-middleware`와 `quipu-server`에 있습니다.

## 서명된 무결성 체크포인트

레코드별 해시 체인은 부분 변조를 잡아냅니다. 레코드를 그 자리에서 고쳐 쓰거나, 세그먼트를 바꿔치기·재배열하는 공격이죠. 하지만 **전체 재작성**은 못 잡습니다. 디스크 권한을 가진 내부자가 세그먼트를 전부 지우고 처음부터 자기모순 없는 체인을 다시 만들어 넣으면, 스토어 안에는 그 위조를 반박할 근거가 없습니다.

체크포인트가 이 빈틈을 메웁니다. 체크포인트는 로그 체인의 상태를 고정합니다.

| 필드 | 의미 |
|---|---|
| `created_at` | 서명 시각 (UTC 마이크로초) |
| `segment_seq` | 체크포인트 시점에 쓰이던 로그 세그먼트 번호 |
| `record_count` | 디스크 위 로그 레코드 수 (보존 정책 적용 후엔 줄어듦) |
| `chain_head` | 가장 최근 로그 레코드의 체인 값 |
| `signature` | 위 필드들에 대한 RSA PKCS#1 v1.5 / SHA-256 서명 |

이것을 스토어 루트의 `checkpoints.log`에 덧붙입니다. 세그먼트 파일은 손대지 않으므로, 체크포인트를 쓰지 않는 스토어는 디스크 포맷이 그대로입니다.

### 체크포인트가 기록되는 시점

- **세그먼트 봉인(seal) 시.** 봉인은 체인의 한 구간이 불변이 되는 순간입니다. 빈도가 `max_segment_bytes`로 제한됩니다. flush/sync 시점을 택하면 N회 append마다 도는 핫패스에 RSA 서명 연산이 올라갑니다.
- **보존 정책으로 세그먼트를 지운 직후.** 직전 체크포인트가 방금 unlink된 세그먼트를 가리킬 수 있습니다. 다시 체크포인트를 찍어, 검증이 정당하게 삭제된 레코드에 의존하지 않게 합니다.
- **수동으로** `AuditStore::checkpoint()` 호출 — 스케줄러에서, 또는 백업 직전에.

### 검증

`verify_integrity()`는 체인 검증을 마친 뒤, 체크포인트가 있으면:

1. 모든 체크포인트의 서명을 RSA 공개키로 검증하고,
2. **가장 최근** 체크포인트의 `chain_head`가 로그 체인 안에 (레코드의 체인 값 또는 세그먼트 시드로) 아직 있는지 확인합니다.

재작성된 체인은 원본 레코드 없이는 체크포인트된 헤드를 재현할 수 없습니다. 꼬리를 잘라낸 체인은 헤드를 잃습니다. 둘 다 이제 검증에서 걸립니다.

최근 헤드만 대조합니다. 오래된 헤드는 보존 정책이 정당하게 지운 세그먼트 안에 있었을 수 있으니까요. 레코드 수가 줄어드는 모든 작업이 새 체크포인트를 남기므로, 최신 체크포인트는 삭제된 데이터에 의존하지 않습니다.

### 쓰기 전용 구성

서명에는 RSA **개인키**가 필요합니다. 공개키만 들고 도는 로그 생산 서비스(권장 구성 — 필드를 암호화할 수는 있지만 되읽지는 못함)는 서명할 수 없습니다. 그래서 거기서는 체크포인트 기능이 **조용히 비활성화**됩니다. `checkpoint()`는 `Ok(None)`을 돌려주고, 세그먼트 봉인 시에도 건너뛰며, `checkpoints.log`는 생기지 않고, `verify_integrity()`는 검사할 체크포인트가 없을 뿐입니다.

에러가 아니라 의도된 선택입니다. 쓰기 경로의 가용성이 먼저니까요. 체크포인트는 개인키가 있는 쪽에서 돌리거나, 쓰기 전용 노드에서는 체인 수준의 변조 증거만으로 운영합니다.

### 외부 앵커링

스토어 안의 체크포인트는 결국 스토어와 운명을 같이합니다. 이 방식은 내부자가 서명 키를 쥐고 있지 않다는 전제 위에 서 있습니다. 키를 가진 내부자라면 재작성한 체인에 새로 서명하면 그만이죠. 앵커 훅은 각 체크포인트를 내부자가 손댈 수 없는 곳으로 내보냅니다.

```rust
let cfg = StoreConfig::new("/var/audit")
    .keys(keys)
    .anchor(|cp| {
        // (created_at, cp.chain_head_hex())를 호스트 밖 어디로든:
        // 다른 머신, 티켓, 투명성 로그, 인쇄된 보고서
    });
```

훅은 체크포인트가 디스크에 기록된 직후 동기로 호출됩니다. 훅 안의 에러와 패닉은 삼켜집니다. 쓰기 경로의 가용성이 앵커링보다 우선이라, 전달 보장(큐잉, 재시도)은 훅 쪽의 몫입니다. 나중에 앵커된 헤드들을 `checkpoints.log`와 대조하면, 체크포인트 파일 자체가 재작성되지 않았음을 증명할 수 있습니다.

### 위협 모델 요약

| 공격 | 탐지 수단 |
|---|---|
| 레코드 in-place 수정 (CRC를 고쳐도) | 해시 체인 |
| 세그먼트 제거 / 재배열 / 교체 | 세그먼트 간 시드 연결 |
| 체인 전체 삭제 후 재작성 | 최신 체크포인트 헤드 |
| 최신 레코드 꼬리 절단 | 최신 체크포인트 헤드 |
| 체크포인트 파일 위조·수정 | RSA 서명 |
| 키 보유자의 체크포인트 파일 재작성 | 외부 앵커 대조 |
